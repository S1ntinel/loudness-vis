# 开发日志

## demo 0.2

- 版本代号：`demo 0.2`
- 记录日期：`2026-05-07`
- 版本定位：在 demo 0.1 的结构基础上，把分析体验做透 + 把多轨道工作流跑通

### 版本摘要

`demo 0.1` 解决了"能跑"，`demo 0.2` 解决"看得清楚、能编辑、能对比"。整体没有改架构（仍然是 React + Vite + Electron + 可选 UV 启动器），而是围绕"分析"和"录音"两个 Tab 把日常使用闭环。

主要方向：

- 分析 Tab 多了**频谱图**和**声场分析球**两个视图，原有面板都补齐了准星/缩放/拖动播放头交互
- 录音 Tab 从单一录音列表升级为**多轨道编辑**：录音 + 上传统一为 Track，能勾选混音、双端点截取片段、试听片段、导出 WAV
- 染色和 LUFS 数据补充到位：响度战争音频也能拉出色差；LUFS 显示 5 项（M / S / I / TP / LRA）
- Electron 侧新增**窗口置顶图钉**，长开仪表盘场景更顺手

### 阶段进度

- [x] 阶段 0：`Vite + React + TS + Electron` 脚手架
- [x] 阶段 1：迁移现有分析大盘到 React
- [x] 阶段 2：录音面板（需求 6）
- [x] 阶段 3：波形多频段染色 + LUFS（需求 1）
- [x] 阶段 4：多音轨导入 + 频响重叠对比（需求 2，通过录音 Tab 多轨 + 频响图叠加联合实现）
- [ ] 阶段 5：系统设备 + 应用音量控制（需求 4，需要 Electron）
- [x] 阶段 6：声场分析球面板（需求 3）
- [ ] 阶段 7：MV 编辑器 Tab（需求 5）

### 这一版的新功能

#### 分析 Tab

- **波形多频段染色**支持三种模式（单色 / RGB 三频段叠加 / 频谱重心 hue 映射），后两种基于 STFT 计算，对响度战争音频也能拉出可读的色差
- **LUFS 数显从 2 项扩到 5 项**：Momentary（400ms）/ Short-term（3s）/ Integrated（整曲，EBU R128）/ True Peak（4× 上采样后的 dBTP）/ Loudness Range
- **频响曲线支持多轨叠加对比**：标题栏的「+ 加入对比」按钮，每条对比通道独立颜色 + 半透明描边 + 可隐藏/移除
- **新增频谱图面板**：整曲 STFT 时频热图，Magma 调色板，时间-频率二维呈现
- **频谱图加准星和拖动播放头**：与频响曲线一致的十字浮窗（显示时间 / 频率 / dB），与波形面板一致的拖拽进度条 + 红色光晕
- **波形 / 频谱图时间轴同步缩放**：Shift + 滚轮以鼠标位置为锚点放大；两个面板同步；双击重置
- **波形 / 频谱图之间加可拖拽分隔条**：自由调整两面板高度比例，比例记忆到 localStorage
- **声场指示器加"球面"模式**：俯视图显示头部 + 4 频段彩色球（低 / 中低 / 中高 / 高），球的位置由实时 L/R 平移决定，大小由能量决定，原 Goniometer 散点保留作为切换项

#### 录音 Tab

- **统一为多轨道编辑器**：录音和上传合并为 `Track` 数据结构（含 audioBuffer / peaks / trim 范围 / 选中状态）
- **拖入和工具栏选文件 → 弹 Modal 选目标**（送入分析 / 送入轨道）；录音 Tab 内的「+ 上传音频」按钮不弹 Modal，直接入轨
- **每条轨道带迷你波形 + 双端点截取**：左右黄色把手拖端点、中间区域拖整段、双击重置；导出片段会按 trim 范围切片成 WAV
- **试听片段 toggle**：单击「▶ 试听 / ⏸ 暂停 / ▶ 继续」，复用一个全局播放器
- **多轨混音**：左侧复选框勾选 N 条轨道，列表头「▶ 播放选中 (n)」一键混音，同样支持暂停/继续，或用「■ 停止」结束
- **录音区比例调整**：上方录音控制压缩到 180px，下方轨道列表吃满剩余空间

#### 顶部工具栏

- **Electron 窗口置顶图钉**：📌 按钮，调用 `BrowserWindow.setAlwaysOnTop`，浏览器环境自动隐藏（依赖 `window.electronAPI` 探测）
- **深 / 浅主题切换**保留，状态记忆到 localStorage

### 当前交互决策记录

延续 demo 0.1 的方向，本版本中确定的交互细节包括：

- 上传文件后弹 Modal 让用户选「分析 / 轨道」，避免与现有"拖入加载到分析"动作冲突
- 双端点截取：端点能拖、中间也能拖整段，双击波形重置
- 试听用独立按钮 toggle，不与拖动播放头共用同一手势
- Shift + 滚轮缩放，锚点为鼠标位置，避免和正常滚动冲突
- 染色 RGB 模式直接把高/中/低频段对应到 R/G/B 通道，让"既有低频又有高频"的瞬间可读
- LUFS 显示横排 5 项，避免占用过多垂直空间

### 当前版本已知边界

- 阶段 5（系统设备 + 应用音量控制）尚未开始，依赖 Electron 主进程接 Windows COM API
- 阶段 7（MV 编辑器 Tab）尚未开始，预计是独立路由 + Canvas 编辑栈 + 视频导出，规模较大
- 多轨道目前只在录音 Tab 编辑/试听，分析 Tab 的多曲线对比仍以静态平均频谱形式呈现，未做"播放期间多轨实时分析"
- True Peak 是 4× 线性上采样的近似版本，不是 sinc 插值，跟专业母带工具会有 ±0.5 dB 偏差
- 频谱图分辨率固定（1500 时间帧 × 512 频段 bin），后续可考虑做"按缩放级别动态加细"
- 频谱图准星显示的 dB 是 `dbFloor + v/255 × (dbCeil - dbFloor)` 反映射，跟实际 STFT 输出的 dB 有量化误差

### 下一阶段建议

如果延续现有路线，下一轮顺序仍按用户在 demo 0.1 决定的"4 → 5 → 3 → MV"，目前 4 与 3 已完成，下一步：

1. 阶段 5：系统设备 + 应用音量控制（Electron 主进程接 Windows COM `IAudioSessionManager2`）
2. 阶段 7：MV 编辑器 Tab（独立模块，建议直接拆 React Router）

- Release 资产为 `UV.zip`（包含 React 构建产物 + UV 本地启动包，解压后可用 `start-demo.cmd` 直接启动本地演示）

## demo 0.1

- 版本代号：`demo 0.1`
- 记录日期：`2026-05-07`
- 版本定位：首个可公开展示的演示版本

### 版本摘要

`demo 0.1` 的重点不是把所有功能一次做完，而是先把项目的主结构、分析体验和演示分发路径搭起来，形成一个可持续迭代的基线版本。

当前版本已经具备以下基础：

- React Web 主界面已建立，可作为后续主开发分支
- 保留 `legacy.html`，并整理出可直接分发的 Lite HTML 包
- 已提供 UV 本地启动包，方便本地演示和测试分发
- Electron 脚手架已接入，为后续设备侧与系统级能力预留入口

### 阶段进度

- [x] 阶段 0：`Vite + React + TS + Electron` 脚手架
- [x] 阶段 1：迁移现有分析大盘到 React
- [x] 阶段 2：录音面板（需求 6）
- [x] 阶段 3：波形多频段染色 + LUFS（需求 1）
- [ ] 阶段 4：多音轨导入 + 频响重叠对比（需求 2）
- [ ] 阶段 5：系统设备 + 应用音量控制（需求 4，需要 Electron）
- [ ] 阶段 6：声场分析球面板（需求 3）
- [ ] 阶段 7：MV 编辑器 Tab（需求 5）

### demo 0.1 已完成内容

#### 1. 主体架构完成切换

- 项目已经从单页 HTML 试验状态进入 `React + TypeScript + Vite` 主开发结构
- Electron 主进程与预加载入口已经接入，但仍处于后续能力承接阶段
- 旧版 `legacy.html` 没有被丢掉，而是作为 Lite 演示分支继续保留

#### 2. 分析主界面可用于演示

- 已完成原有分析大盘向 React 版本迁移
- 保留了波形、频谱、响度相关指标、声场可视化等核心观察路径
- 当前版本已经可以作为“响度战争可视化”主题的演示基础

#### 3. 录音面板已接入

- 已完成录音面板基础接入
- 这使项目不再局限于纯文件分析，也为后续设备输入和系统级采集留出了演进空间

#### 4. 响度与波形表现增强

- 已完成波形多频段染色
- 已加入 LUFS 相关能力
- 这部分是当前 `demo 0.1` 最接近产品价值的可视化增强之一

#### 5. 演示分发路径已打通

- 已提供本地 localhost 启动方式，方便开发和调试
- 已整理 `lite/` 独立 HTML 分发目录，适合轻量演示
- 已整理 `UV/` 本地启动包，适合需要稳定启动流程的测试和内部分享
- GitHub 公共仓库、MIT 协议和基础 README 结构已完成

### 当前交互决策记录

以下内容来自本阶段已明确的交互方向，用于约束后续实现：

- 上传音频文件的入口需要避免与现有“拖入加载到分析面板”产生冲突
- 上传完成后，后续动作应该继续交由用户主动选择，而不是强行打断当前流程
- 拖拽交互范围建议同时支持端点区域和中间区域
- 截图反馈更适合使用“像个简单预览按钮”的形式，而不是过于隐蔽的触发方式

### 当前版本已知边界

`demo 0.1` 仍然是演示版本，不应被描述为完整产品。当前边界主要包括：

- 多音轨导入与频响重叠对比尚未完成
- 系统设备与应用音量控制尚未完成，这部分后续依赖 Electron
- 声场分析球面板尚未完成
- MV 编辑器 Tab 尚未开始
- Electron 当前主要承担结构预留作用，还不是成型桌面版

### 下一阶段建议

如果延续现有路线，下一轮最值得优先推进的是：

1. 阶段 4：多音轨导入 + 频响重叠对比
2. 阶段 5：系统设备 + 应用音量控制
3. 阶段 6：声场分析球面板

原因很简单：这三项比单纯继续修饰界面，更能把项目从“可看”推进到“可比、可控、可扩展”。
